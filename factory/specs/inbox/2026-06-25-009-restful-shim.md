# RESTful Shim Over The Native Action Layer

## Grill Gate

- Decision: internal API remains action-oriented.
- Decision: RESTful endpoints are a shim over actions/projections, not a second
  source of business logic.
- Decision: add this after native actions, tasks, settings, and projections are
  stable.

## Goal

Expose a conventional RESTful client surface for integrations that expect
resource-style endpoints, while preserving the action bus as the execution
authority.

## Scope

This slice owns a small RESTful shim and docs. It must call the same command
bus/reducers/projections as the TUI and native action API.

## Acceptance Criteria

- Define RESTful routes in `API.md` before implementation.
- Add resource endpoints only where the native action/projection layer is
  already stable, likely:
  - runners
  - models
  - chat sessions
  - tasks
  - settings
- Mutating RESTful routes dispatch native actions internally and return
  action/task/revision metadata where useful.
- Read routes use the same projections as `/g0litellama/v1/state`.
- Error responses are stable JSON and documented.
- Add tests proving RESTful mutations produce the same state changes as native
  actions.
- OpenAI `/v1/*` proxy behavior is unchanged.

## Out Of Scope

- Replacing `/g0litellama/v1/actions`.
- Full public API version negotiation.
- Generated clients.
